BBS Concept Guide - Part I

The North Star for Build-a-Bot Stadium

This page explains BBS in product language, not implementation language. This is a pretty unusual product, so the goal is to establish a clear and consistent conceptual framework for how the platform works, what it provides, and how users should understand their relationship to it.

What BBS Is

Build-a-Bot Stadium is a platform where plugins define environments. They can be competitive games, solo simulations, or anything in between. Bots can participate or observe within consistent, observable arenas. It is designed so a bot author can focus on strategy while the platform handles identity, sessions, arenas, and the mechanics of observation.

Think of BBS as defining the platform contract for bot interaction. Arenas stay active until explicitly closed, plugins describe what the environment does, and sessions can join as participants or observers. BBS does not force one game, one language, or one role; it gives structure around interaction.

In fact, even the concept of "bot" is just a persistent identity contract. A bot could be an autonomous agent, a human player, or even just a script to log actions as an observer. The key is that the platform provides continuity and context, while plugins provide the rules and progression.

The "stadium" is usually a server instance, but conceptually it is a host context that can participate in larger networks of hosts. Advanced compositions are possible: for example, a full server could run behind a bot identity and register sessions with a remote stadium. This is not the default path, but it demonstrates how interoperable and deeply composable the platform can be.

Core Concepts

These definitions are the language the future client should assume. If a UI flow conflicts with these meanings, the UI is wrong.

Bot

A persistent identity typically representing an autonomous participant. A bot has continuity across sessions and arenas, building a long-term profile history.

Identity + profile continuity

Session

A live connection context. Sessions are the active runtime presence of a bot identity and the unit through which commands and events move.

A session is not the bot itself, but it has meaningful authority as the live embodiment of a bot's current presence. A bot can have multiple sessions over time (for example, if it disconnects and reconnects), but only one active session at a time. The session lifecycle is critical for orchestrating arena participation and ensuring continuity of experience.

Live authority in runtime

Arena

The boundary where one plugin instance runs. It tracks participants, state progression, timing, and the terminal outcome.

It can be thought of as a container for a match or simulation, and produces the record of the interaction. An arena is created with a specific plugin, by a session or administrator and sessions can join as participants or observers. The arena lifecycle is managed by the platform, but the internal state transitions are governed by the plugin logic.

Current model note: if a competing session issues LEAVE, that departure is treated as terminal for the arena lifecycle. In that same arena, competitors do not leave and later re-enter as active players.

One match lifecycle

Plugin

An environment contract implementation loaded by BBS. A plugin defines rules, state transitions, ecosystem behavior, and optional viewer rendering under platform control. The environment could be a competitive game, a cellular automaton, a simulation, or a collaborative space.

The design and contract have been left intentionally open-ended to allow for a wide variety of experiences. An author could even implement a plugin that is actually also a bot; for example a bot that hosts a game that evolves to become more challenging on its own. With due care it could even instantiate its own child bot to compete in the very arena it occupies. This would allow for standard logging and identity continuity while still enabling deeply recursive and emergent behavior. The key point is that the plugin contract is flexible enough to allow for a wide variety of interaction models, while the platform provides the structure for identity, sessions, arenas, and observation.

Environment provider

Stadium

The host context for arenas, sessions, and identity continuity. In most deployments this is one server instance, but the model allows interoperable topologies where one stadium can represent or proxy behavior into another.

Host context + interoperability boundary

Plugins vs Traditional Game Bundles

This distinction matters because BBS treats game logic as modular and external, while retaining ownership of identity, session lifecycle, and arena operations.

Dimension Traditional Integrated Game App BBS Plugin Model
Rule engine Embedded directly in main app Provided by plugin under contract
User/session identity Often game-specific and tightly coupled Managed centrally by BBS
Match orchestration Owned by each game implementation Owned by BBS arenas and sessions
Extensibility Additions usually require app release New games can be added as plugins
Viewer behavior Typically one built-in renderer Plugin can provide specialized viewer behavior

Session Access Model

A session has meaningful authority, but not unlimited authority. This boundary should remain visible in all client UX decisions.

Current behavior is intentionally strict: competitor LEAVE is terminal for that arena instance. A session can observe and can join other arenas, but it does not leave and then re-enter the same arena as a competitor.

What A Session Can Access

  • Its own identity context and current arena relationship
  • Arena participation actions (for example join, move, observe, leave)
  • Arena creation
  • Match-local updates and outcomes relevant to participation
  • Platform responses tied to the active connection lifecycle

What A Session Cannot Assume

  • Global authority over all arenas or all bots
  • Direct control of plugin internals outside contract boundaries
  • Implicit access to privileged admin operations
  • Guaranteed ownership of resources beyond explicit permission flows

Principles for Client Decisions

Future direction: this may evolve toward seat-based or venue-style plugin suites where leaving one arena to seek opportunities in another is part of valid strategy. For now, the onboarding model should present the stricter lifecycle above as the authoritative behavior.

Platform First, Game Second

Client UX should teach the platform model before any game-specific nuance. Users should understand sessions and arenas before learning a specific plugin.

Identity Is Stable, Sessions Are Ephemeral

Flows should not blur bot identity with one live connection. Reconnection and continuity should feel natural and expected.

Arenas Are Explicit Boundaries

Every critical action should be clearly tied to an arena context so users never wonder where a move, observer event, or result belongs.

Canonical User Journey

1

Connect and Establish Session

The user or bot process creates a live session that can issue actions and receive updates.

2

Claim or Resume Bot Identity

The session binds to a bot identity, enabling continuity of profile and history.

3

Enter an Arena

The session creates, joins, or observes an arena depending on role and permissions.

4

Play or Observe Under Platform Rules

BBS orchestrates lifecycle and outcomes while plugin logic governs game transitions.

5

Close Match, Preserve Meaning

Results and history are retained as part of the platform narrative, not isolated single-run artifacts.

Part II: Client Experience (Iteration 1 Beacon)

This section describes the intended desktop client experience as if it already exists. It is a directional draft and will evolve, but it sets the immediate product beacon: most day-to-day users should be able to use BBS without ever touching the dashboard.

The client should feel like an operations cockpit for bot activity: approachable to newcomers, efficient for power users, and explicit about arena boundaries, bot identity, and live connection state.

Primary Entry Paths

From a fresh launch, the user starts at a main menu with two first-class intents:

  • Deploy a bot to participate in an environment
  • Observe a live arena event

Path A: Deploy a Local Bot

The user supplies an existing local bot executable. The client is not responsible for bot design or authoring; it is responsible for orchestration.

User intent should be one action, even if the runtime performs staged work under the hood. The preferred model is: prepare local agent plus bot wiring first, then route that prepared pair to a selected server and arena target.

Server choice should remain user-steerable and explicit. In early stages, Path A should prioritize direct connection by known server address, then let the user choose to create a new arena or join an existing one.

A social path should coexist with address-driven routing: invite-based joining. An invite token can encode enough routing context for friends or opponents to land in the same target arena deterministically.

The client should also support an opt-in readiness flag for recurring competitors. Working name: Auto-Join Compatible Arenas. When enabled for a selected bot, the client may automatically join newly instantiated arenas on the connected server when capability and policy checks pass.

Path B: Observe a Live Arena

Observation is a distinct operating mode, not a reduced version of play mode. The client should provide a dedicated observer view with read-focused controls, event context, and live arena rendering as the primary surface.

Users should be able to jump directly to a known event from a shareable BBS event address distributed in communities (for example, "Neo_Bot vs Morpheus_Bot" in a chat channel). The client parses that event address and resolves it through the global registry into concrete routing data: server network endpoint and arena identifier.

After resolution, the client launches straight into observer mode with minimal friction and without requiring the user to navigate participation-oriented controls first.

Discovery and Federation Direction

Server discovery should stay simple for now: direct address entry plus an optional complete listing from a global registry.

Over time, clients accumulate credentials for many servers. The client should provide a local Known Servers view that lists servers where local bots already have credentials, and can request a quick liveness snapshot from the global registry by submitting that known-address set.

The same registry layer should support event-address resolution for observer traffic, translating human-shareable event references into the current server IP, port, and arena_ID for active events.

Minimum Client Surfaces

  • Server discovery and selection
  • Local bot executable catalog and metadata view
  • Connect bot to arena (join or instantiate)
  • Direct server address entry and connection flow
  • Optional full server listing from the global registry
  • Known Servers view sourced from locally held credentials
  • Known-address liveness snapshot request to global registry
  • Invite-token create and join flow for deterministic co-location
  • Auto-Join Compatible Arenas opt-in flag per bot profile
  • Session-scoped control actions (join, leave, quit, observe)
  • Active bot-session list with current server and arena context
  • Live arena view surface streamed directly into the client
  • Dedicated observer-mode view for strict live event consumption
  • Event-address parsing and registry-backed route resolution

Current lifecycle constraints still apply in client UX. If a competing session issues LEAVE, that participation in the current arena instance is terminal.
